# Вимоги: Українська постквантова криптографія (ДСТУ-ПК 2026)

> Версія: 1.0
> Статус: APPROVED
> Останнє оновлення: 2026-04-27

## Постановка проблеми

Розвиток квантових обчислень створює загрозу для традиційних криптографічних алгоритмів. Алгоритми Шора та Гровера здатні зламати RSA, ECDSA та інші широко використовувані схеми. Україна потребує власного національного стандарту постквантової криптографії для захисту державних та комерційних комунікацій.

**Чому це важливо:**
- Забезпечення довгострокової безпеки державних комунікацій
- Захист критичної інфраструктури від майбутніх квантових атак
- Збереження криптографічної незалежності та національної ідентичності
- Сумісність з існуючими українськими криптографічними стандартами (ДСТУ 7624, ДСТУ 7564)

## Історії користувачів

### Основні

**Як** системний адміністратор VPN-сервера
**Я хочу** налаштувати підтримку української постквантової криптографії
**Щоб** забезпечити захист від квантових атак при збереженні сумісності з існуючими клієнтами

**Як** розробник клієнтського застосунку
**Я хочу** мати бібліотеку з реалізацією українських криптографічних алгоритмів
**Щоб** інтегрувати національні стандарти у свої продукти

**Як** фахівець з кібербезпеки
**Я хочу** використовувати гібридний режим TLS (класика + постквант)
**Щоб** мати запас міцності навіть при виявленні вразливостей в нових алгоритмах

### Додаткові

**Як** адміністратор PKI
**Я хочу** випускати сертифікати з подвійним підписом (ДСТУ 4145 + Сокіл)
**Щоб** забезпечити плавний перехід на постквантову криптографію

## Критерії приймання

### Обов'язкові (Must Have)

1. **KEM "Мальва" (Malva)**
   **Дано** TLS 1.3 з'єднання з налаштованим UA провайдером
   **Коли** клієнт ініціює handshake
   **Тоді** використовується гібридний KEM (X25519 + Malva-1024) для узгодження ключів

2. **Цифровий підпис "Сокіл" (Sokil)**
   **Дано** сертифікат з ключем Sokil
   **Коли** сервер автентифікується
   **Тоді** клієнт успішно верифікує підпис згідно з алгоритмом на базі решіток

3. **Шифрування "Калина-ПК" (Kalyna-PQ)**
   **Дано** узгоджений сеансовий ключ
   **Коли** передаються дані
   **Тоді** використовується Kalyna-512-GCM для аутентифікованого шифрування

4. **Хешування "Купина-ПК" (Kupyna-PQ)**
   **Дано** потреба в обчисленні хешу
   **Коли** викликається PRF або HMAC
   **Тоді** використовується Kupyna-512 для квантової стійкості

5. **Реєстрація провайдера**
   **Дано** конфігурацію з `cipherSuites: "ua"`
   **Коли** сервер стартує
   **Тоді** UA провайдер реєструється та налаштовує TLS

### Бажані (Should Have)

1. **Гібридні сертифікати**
   - Подвійний підпис: ДСТУ 4145 (512-біт крива) + Сокіл
   - SubjectPublicKeyInfo містить обидва ключі

2. **Автоматичний вибір сертифіката**
   - CertificateStore визначає UA провайдер за типом ключа
   - Функція `IsUACryptoSuite()` для детекції

3. **Конфігурація cipher suite**
   ```go
   const (
       TLS_UA_KALINA_512_GCM_KUPYNA_512 uint16 = 0xUA01
   )
   ```

### Не включено (Won't Have - ця ітерація)

1. **SM9-подібна identity-based криптографія** - може бути додана пізніше
2. **Сертифікати АЦСК з постквантовими ключами** - потребує узгодження з регуляторами
3. **Повна сумісність з ДСТУ 4145** - фокус на нових постквантових алгоритмах
4. **GUI для управління ключами** - лише CLI/API

## Обмеження

- **Технічні**:
  - Реалізація повинна слідувати архітектурі існуючих провайдерів (CN, RU, UK)
  - Тести повинні покривати всі криптографічні примітиви
  - Код має бути Go 1.21+

- **Продуктивність**:
  - Handshake не повинен перевищувати 500ms на сучасному апаратному забезпеченні
  - Накладні витрати шифрування < 5% пропускної здатності

- **Платформа**:
  - Linux AMD64 (основна)
  - macOS ARM64 (розробка)
  - Windows AMD64 (підтримка)

- **Залежності**:
  - Базується на існуючих реалізаціях Калини та Купини (ДСТУ 7624, ДСТУ 7564)
  - Не потребує зовнішніх криптографічних бібліотек
